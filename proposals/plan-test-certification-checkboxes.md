## Problem

Plans can be marked complete without evidence that work was verified. Three
prior designs — the `## Testing Plan` section, and two proposed checkboxes
(`tests_defined`, `tests_passed`) — were critiqued and found to share the
same fundamental flaw: they are **checkboxes that rely on human discipline**.

Research across regulated industries (DO-178C, 21 CFR Part 11), software
supply chain security (SLSA, in-toto, sigstore), and AI safety governance
produced a clear consensus:

> *"The one thing that definitively does not work: adding more checkboxes.
> Every mechanism shifts the rubber-stamp problem rather than eliminating it.
> The honest goal is to shift rubber-stamping to a place where it is
> (a) harder, (b) detectable, and (c) attributable."*

The solution is to replace self-reported attestation with **machine-produced
committed evidence** wherever possible, and to make human judgment traceable
and attributed where it remains necessary.

---

## Proposed Solution — Two Tiers

### Tier 1: Machine-produced evidence (tests_passed → committed artifact)

**`tests_passed` is not a checkbox.** It is a committed `## Test Results`
section produced by the test runner and validated by the pre-commit hook.

When tests run, the test runner appends or replaces the Test Results section:

```markdown
## Test Results

<!-- generated by scripts/run-plan-tests.js — do not edit manually -->
Run date: 2026-06-04T14:23:00Z
Commit: a4b2c9f
Result: ✅ 9/9 passed

| # | Test | Result | Output |
|---|------|--------|--------|
| 1 | Hook blocks pending steps | ✅ Pass | 9/9 cases |
| 2 | Clean commit passes hook | ✅ Pass | exit 0 |
```

The pre-commit hook validates when `status: completed` is being set:
1. The `## Test Results` section exists in the plan body
2. The run date is within the last N commits (not stale)
3. All table rows contain ✅ Pass (no ❌ or ⏳ rows)

No human clicks `tests_passed`. The runner produces the evidence or it does
not exist. A human cannot fabricate the section without it being immediately
obvious as a manual edit (the comment warns; the hook checks timestamps).

This draws directly from in-toto/DSSE attestation patterns: the artifact
*is* the evidence, not a claim about the evidence.

### Tier 2: Human judgment with structural validation (tests_defined)

**`tests_defined` remains a human checkbox** — because adequacy of a test
suite requires judgment that cannot be automated. But it is coupled with
structural constraints that make rubber-stamping harder and detectable:

```yaml
tests_defined: false   # human certifies: "this suite would catch regressions"
```

**Structural constraints enforced by hook and critique tool:**
- Test row count ≥ deliverable row count (structural minimum)
- Each test row must name a specific command or acceptance criterion
- Each test row must reference which deliverable it verifies
- If deliverable count has grown since `tests_defined` was last set to `true`,
  it is automatically reset to `false` (cascade enforced by hook)

**PropertiesPane tooltip (what the human is certifying):**
> "This test suite would catch regressions if the implementation were wrong."

Not "tests exist." Adequacy, not existence.

### Independence requirement (structural, not procedural)

Drawn from DO-178C: **the plan author should not be the sole certifier.**

In the frontmatter:
```yaml
assigned_to: NetYeti        # developer/implementer
gate_reviewer: NetYeti      # SHOULD differ from assigned_to on team projects
```

The hook warns (not blocks, for solo projects) when `gate_reviewer` equals
`assigned_to`. On team projects this becomes a block. The audit trail always
records who checked `tests_defined` — attributing the certification, not just
the action.

### Challenge-response for tests_defined

Instead of just a checkbox, when the human checks `tests_defined` in the
Web UI, they are prompted with one question drawn from the test suite:

> "Which deliverable does test #3 verify?"

The answer is stored in the audit log alongside the certification timestamp.
A reviewer who cannot answer has not read the test suite. A reviewer who
types gibberish is distinguishable from one who answers correctly. The answer
is not verified for correctness — it is recorded for accountability.

## What each mechanism resolves

| Prior failure mode | Mechanism | Genuinely resolved? |
|---|---|---|
| Rubber stamp on tests_passed | Machine-produced artifact (runner commits results) | ✅ Yes — no checkbox to click |
| AI-generated trivial tests | Each test must reference a deliverable; adequacy certification | ⚠️ Shifted — to: are tests well-scoped? |
| Staleness of tests_defined | Hook resets when deliverable count grows | ✅ Yes — structural, not convention |
| All-or-nothing masking failures | Test Results table has per-row status; hook checks all rows | ✅ Yes — row-level granularity enforced |
| Non-code acceptance criteria | Test runner produces acceptance checklist section; same format | ✅ Yes — same mechanism, different content |
| Old Testing Plan was decorative | Test Results section is hook-enforced and runner-generated | ✅ Yes — structural difference |
| AI governance | Committed artifact is not a field AI can set in YAML | ⚠️ Shifted — to: can AI fake the runner output? |
| Rubber stamp on tests_defined | Challenge-response prompt + attribution in audit log | ⚠️ Shifted — to: does person answer honestly? |

**What is genuinely shifted but not eliminated:**
- Test *quality* (tests that run and pass but don't mean anything)
- Human honesty in the challenge-response
- Team independence in solo projects

The research consensus: these are acceptable shifts. Test quality is the
hardest problem in software and is not solvable by governance alone. The
remaining rubber-stamp risk is now attributable, detectable on audit, and
harder to execute unconsciously.

## The test runner

`scripts/run-plan-tests.js` — a tool-agnostic script (like `critique-plan.js`)
that:
1. Reads the plan's `## Tests` table
2. Executes each test command
3. Records per-row results (pass/fail/skip + timing + output snippet)
4. Writes `## Test Results` section back to the plan file
5. Exits non-zero if any test failed

Can be called by Claude, OpenCode, CI, or any AI tool. The hook validates
the output exists; the runner produces it or it doesn't exist.

---

## Migration of existing plans

Existing plans without `tests_defined` or `## Test Results` sections:
- Get `tests_defined: false` added by a migration script
- Are not blocked for completion if `## Tests` section also absent (grace period)
- The grace period ends at Phase 2 start — all active plans must have test
  suites before any Phase 2 plan can be started

## Resolved design decisions

**Required on ALL plans** — yes, with acceptance criteria for non-code
deliverables (documentation, policy, brand assets). The test runner supports
manual confirmation rows alongside executable test rows.

**Unchecking `tests_defined` cascades to clearing `## Test Results`** — yes.
If the test suite changes, previously-produced results are no longer valid.
The hook catches if `tests_defined: false` but `## Test Results` shows pass.

**Critique tool checks staleness** — yes, and generates the initial `## Tests`
table when `tests_defined: false`. The critique tool also checks that test
count ≥ deliverable count on already-defined plans.

## Out of Scope

| Idea | Why deferred |
|------|-------------|
| Cryptographic signing of test results (sigstore/in-toto) | The committed artifact is sufficient for Phase 1; cryptographic signing is Phase 3+ when external deployments need supply-chain proof |
| Mandatory second reviewer (DO-178C independence) | Requires team structure; warn for solo, block for team projects in Phase 3 |
| Random audit of past approvals | Strong mechanism; captured for future implementation |
| Full SOC2/ISO27001 audit trail | Enterprise Phase 4 concern |
