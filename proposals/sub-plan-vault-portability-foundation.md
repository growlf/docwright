## Problem

DocWright currently assumes it lives inside its own repo. Hardcoded paths prevent it from managing external projects. There is no standard way for a vault to declare where DocWright lives, and no template for wiring MCP servers in a vault-scoped way.

## Proposed Solution

Deliver two tightly-coupled changes that together make vaults path-portable:

1. **No hardcoded paths** — `docwright init` writes `.env` with `DOCWRIGHT_PATH` as the first step. All vault-side files (`.claude/settings.json`, hook commands, config references) use `$DOCWRIGHT_PATH` — never absolute paths. Hook install script accepts `--vault /path`.

2. **`.mcp.json` template** — Define a vault-scoped `.mcp.json` that wires two MCP servers: `dw-vault` (vault mode, serves this vault's lifecycle) and `dw-upstream` (upstream mode, targets DocWright's own repo). Both reference `$DOCWRIGHT_PATH`. Template generated by `docwright init`.

## Parent Reference

## Dependencies

- **Prerequisite:** TypeScript MCP Server migration (sub-plan #1) — the TS server must support `--mode` before the `.mcp.json` template can wire it
- **Blocking:** `docwright init` scaffold (sub-plan #3), MSP Pilot (sub-plan #7), Cascade STEAM (sub-plan #8)

## Future

As new MCP modes are added (e.g., `--mode audit`), the `.mcp.json` template expands. The path resolution pattern becomes the standard for all vault deployments.
