## Summary

Root-caused why the urgent GitHub issue #68 (dev-tracking split proposal) kept getting missed at session start across three consecutive sessions: the docwright-session-start skill is discretionary, only firing on phrase-matches like 'resume', so a session that opens with an unrelated task (as this one did, with the devdw SSH question) never triggers it and never sees the urgent banner. Fixed structurally per this repo's code-over-memory policy by adding a SessionStart hook (scripts/claude-session-start-hook.sh) that runs unconditionally at the start of every session and injects any open GitHub issues labeled urgent as additionalContext, independent of the user's first message. Verified the hook correctly detects issue #68 via pipe-test and validated the settings.json schema with jq -e; user confirmed the offline/no-gh-fallback case does not need to be hardened further right now. Also merged PR #74 (devdw session note) after branch-protection rejected a direct push to main, and corrected the stale feedback-git-workflow memory that still described the retired GitFlow/develop process. Began working issue #68's open design items (demand-count weighting, beta gate threshold N, milestone naming, Pitfalls/Alternatives pass) but did not reach a conclusion before session end.
