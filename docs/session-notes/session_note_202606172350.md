# Session Note: 2026-06-17 — Roadmap Restructure, Vault Foundation, UI Fixes

**Date:** 2026-06-17
**Author:** NetYeti @ phoenix
**Focus:** Roadmap restructure, vault write API & KG foundation, UI cleanup, bug fixes

## Summary

Major session with two main threads. First: deep architectural rethinking of the
roadmap — vault write API and vault document index moved to Phase 3 (before pilots),
knowledge graph moved to Phase 3/4 (perception before governance enforcement), and
the wikilink index unified with frontmatter edges so the KG is accurate from day one.
The key insight: "completeness is not enrichment — it's correctness." Second: a series
of UI and bug fixes — status page cleanup, dedicated audit page, executor presence
indicator, improve API code-fence stripping, and the walkDeps bug that caused
`related_to` links to be incorrectly consumed as plan dependencies.

## Decisions Made

- **Vault write API (3a) in Phase 3** — `fix-stale-approvals.ts` and `backfill-proposal-source.ts` exist because this API doesn't; pilots will produce broken data without it
- **Vault document index (3b) in Phase 3** — frontmatter edges + wikilink body-text edges in one unified index; no "basic graph now, wikilinks later" split; accuracy is not optional
- **Knowledge graph (3c) in Phase 3** — oversight tool, not a feature; must be visible before pilots run; ships with wikilink edges from day one
- **Phase 4 simplified** — exactly three serial governance items: mode enforcement → profile engine → lifecycle gates
- **walkDeps must only follow `depends_on`** — `related_to` is context, not a bundling dependency; `blocks` likewise excluded
- **Executor presence indicator** — recurring silent-executor confusion; toolbar pill with amber/blue/green states, links to execute panel
