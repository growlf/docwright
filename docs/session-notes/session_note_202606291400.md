# Session Note: 2026-06-29 — ForceGraph Bug Root Cause & Lifecycle Transition

**Date:** 2026-06-29
**Author:** NetYeti @ phoenix
**Focus:** ForceGraph connector pile-up — real root cause + lifecycle docs

## Summary

Resumed after a dropped connection. Found an in-flight lifecycle transition for the
`forcegraph-client-nav-sizing` proposal mid-way through — proposal staged for deletion,
approved copy and plan scaffolded but not committed. Filled in the plan and went to
commit, but discovered the graph bug (lines piling at center on client-side navigation)
still existed despite the v0.4.1 fix in `ForceGraph.svelte`.

Diagnosed the real root cause: `rawEdges` in `KnowledgeGraph.svelte` is a Svelte `$state`
deep reactive proxy. D3's `forceLink` mutates each edge's `source`/`target` field from a
string ID to a node object reference during simulation init. Svelte's proxy intercepts
these writes, invalidates `forceEdges`, and re-fires the `$effect` in `ForceGraph` that
calls `buildGraph()` — stopping the first simulation before nodes could spread from center.
Fixed by shallow-copying edges to plain objects (`rawEdges.map(e => ({ ...e }))`) before
passing them to ForceGraph, so d3 mutations don't touch the reactive state.

Also fixed 3 pre-existing failing dispatch tests: `fix-stale-approvals` tests were passing
`{ ...process.env }` to the spawned script, and `DOCWRIGHT_VAULT_ROOT` was set in the
shell, redirecting the script to the real vault instead of the test's temp directory.

## Decisions Made

- The v0.4.1 `ForceGraph.svelte` change (remove fallback dimensions + `afterNavigate`)
  was correct but only addressed a secondary symptom. The primary bug was the Svelte
  `$state` proxy / d3 mutation loop in `KnowledgeGraph.svelte`.
- Plan `forcegraph-client-nav-sizing` status: `active`, all 3 steps done. Needs human
  approval to be marked `completed`.
- Test env isolation: test runners that spawn sub-processes must scrub
  `DOCWRIGHT_VAULT_ROOT` from the env to prevent real-vault contamination.
