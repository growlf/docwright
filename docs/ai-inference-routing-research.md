### The Real Problem

The Intel Arc integrated GPU has **zero dedicated VRAM**. All model data shares system memory:
- 7B models need ~4-6 GB to load
- 14B models need ~8-16 GB
- With only 7.0 GiB available, 14B models exceed headroom → swap → **GPU falls back to CPU** → massive CPU heat → crash/thermal event

Docker ollama verification from logs:
```
get_memory_info: [warning] ext_intel_free_memory is not supported
load_tensors: tensor 'token_embd.weight' cannot use SYCL_Host, using CPU instead
```

The driver can't properly detect free GPU memory because `ext_intel_free_memory` is unsupported in the Level Zero driver, causing it to report total system RAM (~30 GB) as "free" even when only 7 GB is actually available.

## Action Items

### 1. Fix Olla Routing Priorities

**Problem:** Remote Nvidia node (10.10.0.201) has the same priority (70) as the local Intel Arc Docker. Olla treats them equally.

**Fix:** Bump `10_10_0_201` to priority 90 so all inference routes there first. Local Arc serves as fallback only.

### 2. Understand the `phoenix` Auto-Discovered Endpoint

The Olla status shows 3 endpoints: `ollama` (Docker, 15 models), `phoenix` (host, 22 models), and `litellm-cloud`. The `phoenix` endpoint appears to be auto-discovered from the host's own Ollama service. It has never been used. Needs investigation whether it's redundant or conflicting.

### 3. Intel Arc Memory Optimization

- **Short term:** Cap local Ollama memory to prevent exceeding actual available RAM (see `OLLAMA_MEM_LIMIT` in docker-compose.arc.yml — currently 32g, should be ~8g)
- **Long term:** The integrated GPU cannot handle 14B+ models. Route all heavy inference to the Nvidia node.

### 4. Model Selection for Tool Use

With routing fixed, select the appropriate model:
- **Remote:** `qwen2.5-coder:14b` or `qwen2.5:14b` on the Nvidia node (fast, good tools)
- **Local fallback:** `llama3.1:8b` on Intel Arc (best tool support, 22s latency)
- **Local emergency:** `qwen2.5:1.5b` or `qwen2.5:7b` for lightweight tasks
