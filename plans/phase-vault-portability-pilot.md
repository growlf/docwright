# Phase 3 — Vault Portability, Real-World Pilot & Upstream Contribution Pipeline

## Overview

Establishes clean tool/vault separation so DocWright can manage any external project,
not just itself. Delivers a real-world MSP pilot (non-profit managed services), a
structured upstream contribution pipeline, and the architectural foundation that all
subsequent phases build on.

See [[proposals/approved/phase-vault-portability-pilot.md]] for full design rationale,
architecture diagrams, and alternatives considered.

**Prerequisite:** Phase 2 gate review by NetYeti must complete before this plan begins.

## Implementation Steps

| Step | Action | Details | Status |
|------|--------|---------|--------|
| 1 | Vault-portable TypeScript MCP server | Rewrite `scripts/mcp-server.py` in TypeScript. `DOCWRIGHT_VAULT_ROOT` env var is first-class config. `--mode upstream` flag switches the server to target DocWright's own repo instead of the vault. ## Testing Plan

- TypeScript MCP server: parity test all tools in both vault and upstream modes against the existing Python server baseline
- `docwright init`: end-to-end smoke test — init a fresh vault, open in web UI, complete a proposal→plan→completed cycle
- Profile override merge: unit test all three merge modes (scalar replace, object deep-merge, `+array` append vs array replace)
- `vault:migrate`: test a no-op migration (same version) and a breaking migration (field rename) — vault content must be untouched
- MSP pilot: manual acceptance test — full lifecycle cycle with zero manual file edits
- Contribution pipeline: test with token absent (URL fallback) and token present (issue creation); verify `DOCWRIGHT_CONTRIB_APPROVED=1` gate blocks without env var

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| TypeScript MCP rewrite breaks existing governance flow | Medium | High | Parity test all tools against Python baseline before removing Python server |
| `DOCWRIGHT_PATH` env var not set in vault — hooks silent-fail | Medium | High | `docwright init` sets it; hook scripts check for var and fail loudly with actionable message |
| Profile override merge corrupts bundled profile state | Low | High | Merge produces a new object — never mutates bundled profile in memory or on disk |
| MSP pilot reveals architectural gaps too large to patch mid-phase | Medium | Medium | Friction log captures them; larger gaps become Phase 4 proposals rather than blocking the pilot |
| GitHub token scoping exposes too much vault context | Low | High | Token needs only `issues: write` scope; sanitization schema enforced at MCP layer |
| Phase 2 not fully closed before Phase 3 begins | Medium | Medium | Explicit gate: `research-plan-execution-modes` and `fix-stale-approved-proposals` must complete first |
