# Bug: Plan Completion Gate Enforcement Has Multiple Critical Gaps

## Overview

*Plan generated from approved proposal: Bug: Plan Completion Gate Enforcement Has Multiple Critical Gaps*

### Problem

DocWright's plan completion enforcement has at least four distinct gaps that allow
plans with unfinished work to reach `status: completed`. Discovered during the
AI Task Category Taxonomy plan session (2026-06-17) when the "Complete" button was
visible and presumably clickable on a plan with multiple pending steps, unchecked
gate criteria, and unreviewed tests.

### Gap 1 — `checkCompletionGate` scans for `## Phase Gate`, not `### Gate Criteria`

`src/mcp/lib/steps.ts` line 83:
```typescript
inGate = line.includes('Phase Gate');
```

Every plan auto-generated by `approve-proposal/+server.ts` uses `### Gate Criteria`
(not `## Phase Gate`). The gate check therefore always fires with:

```
ERROR: Plan '...' has no ## Phase Gate section.
```

This is the wrong error. The gate criteria exist — they're just under a different
heading that the validator doesn't recognize. The result is that clicking Complete
on any auto-generated plan produces a confusing error, rather than the correct
"N unchecked gate items" error.

**Affected:** All plans generated by the approve-proposal endpoint (the majority
of plans in the system).

### Gap 2 — `- [ ]` checkboxes in `### Gate Criteria` are never enforced

Because Gap 1 means the gate check never enters the `### Gate Criteria` section,
unchecked `- [ ]` items there are never read. A plan with all gate criteria
unchecked can be completed as long as the Implementation Steps table shows all
✅ Done.

**Affected:** All gate criteria checkboxes in all auto-generated plans.

### Gap 3 — `tests_human_reviewed: false` is not a completion gate

`checkCompletionGate` checks `tests_defined: true` but does NOT check
`tests_human_reviewed: true`. A plan whose test plan was auto-generated and never
reviewed by a human can be completed freely. The `tests_human_reviewed` field
exists in every plan frontmatter but has no enforcement.

**Risk:** Plans complete with unverified AI-generated test definitions, making the
test coverage claim meaningless.

### Gap 4 — Complete button always visible for in-progress plans

The Web UI shows the Complete button solely based on `status: in-progress`. No
pre-flight check for pending steps, unchecked gate criteria, or
`tests_human_reviewed: false`. The human only discovers blockers after clicking.

**Risk:** Creates false confidence that the plan is completeable, and forces
discovery-by-failure rather than proactive guidance.

### Proposed Fix

### Fix 1 — Section name normalization (critical)

Update `checkCompletionGate` to recognize both `## Phase Gate` (legacy) and
`### Gate Criteria` (current template):

```typescript
inGate = line.includes('Phase Gate') || line.includes('Gate Criteria');
```

### Fix 2 — `tests_human_reviewed` as a completion gate (critical)

Add to `checkCompletionGate` after the `tests_defined` check:

```typescript
const testsReviewed = extractFrontmatterField(text, 'tests_human_reviewed');
if (String(testsReviewed) !== 'true') {
  return `ERROR: Plan '${planName}' has tests_human_reviewed=false. A human must review and certify the test plan before the plan can be completed.`;
}
```

### Fix 3 — Disable Complete button client-side when blockers exist (high)

In `PropertiesPane.svelte`, before showing the Complete button as enabled:
- Parse the plan body for `⏳ Pending` rows in the Implementation Steps table
- Parse for unchecked `- [ ]` items in the Gate Criteria section
- Check `tests_human_reviewed` frontmatter field
- Show button disabled with a tooltip listing what's blocking

This prevents discovery-by-failure and matches how the Approve button already
shows a tooltip explaining why it's disabled (though we removed the Improve gate,
the pattern is correct).

### Fix 4 — Update `approve-proposal` template to use `## Phase Gate` (medium)

The legacy section name `## Phase Gate` is what the existing validator understands.
The atomic plan generator and the template fallback should both use `## Phase Gate`
until Fix 1 lands. Alternatively, fix the validator first (Fix 1) then the template
naming is less important.

### Implementation order

1. **Fix 1 + Fix 2** first — both are one-line changes in `src/mcp/lib/steps.ts`.
   These close the critical enforcement gaps immediately.
2. **Fix 3** — Web UI pre-flight. Requires reading plan body client-side before
   enabling the button.
3. **Fix 4** — Template naming cleanup. Lower priority once Fix 1 normalizes the
   heading detection.

### Why this is critical

The plan completion gate is DocWright's primary mechanism for ensuring that:
- All implementation steps were actually completed
- Human review was applied to AI-generated content
- Gate criteria signed off by the right people

All three assurances are currently either broken (Gaps 1, 2) or unenforced (Gap 3).
A plan marked `completed` in DocWright currently provides no governance guarantee
that the work described was actually verified. This undermines the core value
proposition of the governance system.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Heading normalization misses a variant in existing plans (e.g., `## Phase Gate` vs `## Phase Gate `) | Medium | High | Add whitespace-stripping and case-insensitive comparison in the regex; test against all existing plan files before deploy |
| `tests_human_reviewed` field missing from some plan frontmatter causes `checkCompletionGate` to throw instead of failing open | Medium | High | Treat missing field as `false` by default in the gate check; log a warning but do not crash the MCP tool |
| Client-side Complete button state desynchronizes from MCP server state due to stale props | Medium | Low | Add a `disabled` title tooltip showing which gate is blocking; force re-check on every `onMount` and on every plan update event |
| A plan with `## Phase Gate` but no `### Gate Criteria` list passes Fix 1's heading detection prematurely | Low | Medium | Gate check must require both heading presence AND at least one unchecked criteria remain — not just the heading string |
| Contributors update frontmatter manually (bypassing MCP tools) and set `tests_human_reviewed: true` without actually reviewing | Low | High | Document in SOP that `tests_human_reviewed` must only be set via the UI toggle; add a pre-commit hook that rejects manual edits to that field |
| Legacy `approve-proposal` template (Fix 4) uses `## Phase Gate` but other templates still use `### Phase Gate` inconsistently | Low | Low | Run a one-time grep across all `templates/` after Fix 4 lands; fix any stragglers as a follow-up chore |
